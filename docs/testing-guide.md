# 🧪 Посібник з тестування та аудиту платформи G.E.N.A. v6.0.0

Цей документ містить покроковий протокол перевірки (QA Checklist) для підтвердження того, що інфраструктура розгорнута коректно, сервіси взаємодіють між собою, а правила безпеки надійно захищають сервер від зовнішніх загроз.

Використовуйте цей чек-лист після кожного свіжого розгортання платформи.

---

# 🛠 Фаза 1: Базовий рівень (ОС та Kubernetes)

**Мета:** Переконатися, що системні компоненти працюють стабільно і споживають адекватну кількість ресурсів.

- [ ] **Доступ до Cockpit**
  - Відкрийте `https://<IP_сервера>:9090`
  - Перевірте метрики CPU/RAM та перегляньте системні логи на наявність критичних помилок.

- [ ] **Статус кластера K3s**

```bash
kubectl get nodes
kubectl get pods -A
```

Усі поди в просторах імен `kube-system`, `argocd`, `monitoring`, `tekton-pipelines`, `minio`, `registry` мають бути у статусі **Running** або **Completed**.

- [ ] **Перевірка UFW**

```bash
sudo ufw status verbose
```

Має бути увімкнений (**active**) із політикою **Default: deny (incoming)**.

# 🏗️ Фаза 2: Автоматизація CI/CD та GitOps (Tekton + ArgoCD)

**Мета:** Перевірити повний цикл розробки — від автоматичної збірки коду всередині кластера до його розгортання.

- [ ] **Доступ до Tekton Dashboard**
  - Відкрийте `https://<TEKTON_HOST>`
  - Переконайтеся, що інтерфейс доступний, а в розділі Tasks присутні базові таски (`git-clone`, `kaniko`).

- [ ] **Тестування Webhook та автоматичної збірки**
  - Зробіть тестовий пуш у Git-репозиторій.
  - У Tekton Dashboard → PipelineRuns має з'явитися новий процес збірки.

- [ ] **Аудит локального реєстру**

```bash
curl -X GET http://registry.registry.svc.cluster.local:5000/v2/_catalog
```

- [ ] **Створення тестового додатку в ArgoCD**
  - Відкрийте `https://<Ваш_ARGOCD_HOST>`
  - Створіть новий Application.
  - Вкажіть Git-репозиторій та шлях до маніфестів.

- [ ] **Синхронізація (Sync)**
  - Додаток має перейти у статус **Healthy** та **Synced**.

- [ ] **Перевірка в Portainer (опціонально)**
  - Переконайтеся, що новий namespace та контейнери успішно створені.

# 🗄️ Фаза 3: Автономне S3-сховище та резервне копіювання (MinIO + Velero)

**Мета:** Перевірити працездатність in-cluster сховища та готовність системи до резервного копіювання.

- [ ] **Доступ до MinIO UI**
  - Відкрийте `https://<MINIO_HOST>`
  - Авторизуйтеся за допомогою `MINIO_ROOT_USER` та `MINIO_ROOT_PASSWORD`.

- [ ] **Автоініціалізація бакета Velero**
  - Перейдіть до Buckets.
  - Переконайтеся, що існує бакет `velero`.

- [ ] **Зв'язок Velero зі сховищем**

```bash
kubectl logs -n velero deploy/velero
```

Переконайтеся, що немає помилок авторизації або підключення.

---

# 🔒 Фаза 4: Маршрутизація та SSL (Ingress)

**Мета:** Перевірити веб-шлюз та автоматичну генерацію сертифікатів.

- [ ] **Доступність сервісів**

  - `https://tekton.yourdomain.com`
  - `https://minio.yourdomain.com`
  - `https://demo.yourdomain.com`

- [ ] **Перевірка SSL-сертифіката**
  - Сертифікат має бути дійсним.
  - Видавець: Let's Encrypt або Let's Encrypt Staging.

- [ ] **Примусовий HTTPS-редирект**

Перейдіть на:

```text
http://demo.yourdomain.com
```

Очікується відповідь:

```text
308 Permanent Redirect
```

# 📊 Фаза 5: Observability (Grafana + Prometheus + Loki)

**Мета:** Перевірити збір метрик та логів.

- [ ] **Перевірка метрик (Prometheus)**
  - Grafana → Dashboards → Kubernetes / Compute Resources / Cluster
  - Графіки повинні відображати актуальне навантаження.

- [ ] **Перевірка логів (Loki)**

Запити:

```logql
{namespace="tekton-pipelines"}
```

або

```logql
{namespace="minio"}
```

Повинен відображатися потік логів у реальному часі.

# 🚨 Фаза 6: Сповіщення та Signal Webhook

**Мета:** Перевірити роботу Alertmanager та доставку повідомлень у Signal.

- [ ] **Симуляція аварії**

```bash
kubectl run crash-test --image=busybox --restart=Always -- /bin/sh -c "sleep 5; exit 1"
```

- [ ] **Очікування алерта**
  - Зачекайте 2–3 хвилини.
  - Має спрацювати Alertmanager.

- [ ] **Підтвердження**
  - У Signal має надійти повідомлення про аварію.

- [ ] **Очищення**

```bash
kubectl delete pod crash-test
```

Після відновлення має надійти повідомлення `[RESOLVED]`.

# 🛡️ Фаза 7: Аудит безпеки (Security Hardening Test)

**Мета:** Перевірити захист від несанкціонованого доступу.

- [ ] **Заборона доступу по паролю**

```bash
ssh -o PreferredAuthentications=password root@<IP_сервера>
```

Очікується:

```text
Permission denied
```

- [ ] **Захист від брутфорсу (Fail2Ban)**
  - Виконайте 6–7 невдалих спроб входу.
  - IP має бути автоматично заблокований.

Розблокування:

```bash
sudo fail2ban-client set sshd unbanip <ВАША_IP>
```

- [ ] **Сканування портів**

```bash
nmap -Pn -p- <IP_сервера>
```

Повинні бути відкриті лише:

- 22 (SSH)
- 80 (HTTP)
- 443 (HTTPS)
- 6443 (K3s API)
- 9090 (Cockpit)

Інші порти не повинні бути доступні ззовні.
