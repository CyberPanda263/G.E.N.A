# 🧪 Посібник з тестування та аудиту платформи G.E.N.A.

Цей документ містить покроковий протокол перевірки (QA Checklist) для підтвердження того, що інфраструктура розгорнута коректно, сервіси взаємодіють між собою, а правила безпеки надійно захищають сервер від зовнішніх загроз.

Використовуйте цей чек-лист після кожного свіжого розгортання платформи.

---

## 🛠 Фаза 1: Базовий рівень (ОС та Kubernetes)
Мета: Переконатися, що системні компоненти працюють стабільно і споживають адекватну кількість ресурсів.

- [ ] **Доступ до Cockpit:** Відкрийте `https://<IP_сервера>:9090`. Перевірте метрики CPU/RAM та перегляньте системні логи на наявність критичних помилок.
- [ ] **Статус кластера K3s:** Підключіться по SSH та виконайте:
  ```bash
  kubectl get nodes
  kubectl get pods -A
  ```
  *Усі поди в системних просторах імен (kube-system, argocd, monitoring) мають бути у статусі `Running` або `Completed`.*
- [ ] **Перевірка UFW:**
  ```bash
  sudo ufw status verbose
  ```
  *Має бути увімкнений (active) із політикою `Default: deny (incoming)`.*

## 🐙 Фаза 2: GitOps та розгортання додатків
Мета: Перевірити пайплайн безперервного розгортання (CD) через ArgoCD.

- [ ] **Створення тестового додатку:** В UI ArgoCD створіть новий додаток (Application), вкажіть ваш Git-репозиторій та шлях до тестових маніфестів (наприклад, `examples/demo-app`).
- [ ] **Синхронізація (Sync):** Натисніть кнопку Sync. Додаток має перейти у статус `Healthy` та `Synced`.
- [ ] **Перевірка в Portainer (опціонально):** Відкрийте Portainer і переконайтеся, що новий простір імен та контейнери успішно створені та працюють.

---

## 🔒 Фаза 3: Маршрутизація та SSL (Ingress)
Мета: Перевірити роботу веб-шлюзу та автоматичну генерацію сертифікатів Let's Encrypt.

- [ ] **Доступність ззовні:** Відкрийте в браузері домен розгорнутого додатку (наприклад, `https://demo.yourdomain.com`).
- [ ] **Перевірка SSL-сертифіката:** Натисніть на іконку "замка" в адресному рядку браузера. Сертифікат має бути дійсним і виданим *Let's Encrypt* (або *Let's Encrypt Staging*, залежно від вибору при встановленні).
- [ ] **Примусовий HTTPS-редирект:** Спробуйте зайти по HTTP: `http://demo.yourdomain.com`.
  *Nginx Ingress має автоматично повернути `308 Permanent Redirect` на HTTPS.*

## 📊 Фаза 4: Observability (Grafana + Prometheus + Loki)
Мета: Перевірити, чи збирається телеметрія та логи з кластера.

- [ ] **Перевірка метрик (Prometheus):** Відкрийте Grafana -> Dashboards -> *Kubernetes / Compute Resources / Cluster*. Графіки мають показувати реальне навантаження.
- [ ] **Перевірка логів (Loki):** Відкрийте Grafana -> *Explore*. Оберіть джерело `Loki` та виконайте запит `{namespace="argocd"}`.
  *Ви маєте побачити кольоровий потік логів у реальному часі від компонентів ArgoCD.*

---

## 🚨 Фаза 5: Сповіщення та Signal Webhook
Мета: Переконатися, що Alertmanager успішно фіксує проблеми та відправляє їх через Python-адаптер у месенджер Signal.

- [ ] **Симуляція аварії:** Створіть Pod, який постійно перезавантажується з помилкою:
  ```bash
  kubectl run crash-test --image=busybox --restart=Always -- /bin/sh -c "sleep 5; exit 1"
  ```
- [ ] **Очікування алерта:** Зачекайте 2-3 хвилини. Prometheus має зафіксувати стан `CrashLoopBackOff`, передати його в Alertmanager, а той — викликати ваш вебхук.
- [ ] **Підтвердження:** Ви маєте отримати повідомлення в додатку Signal з детальним описом аварії та червоним/жовтим емодзі.
- [ ] **Очищення:** Видаліть тестовий Pod:
  ```bash
  kubectl delete pod crash-test
  ```
  *Після відновлення стану кластера ви маєте отримати повідомлення `[RESOLVED]` із зеленим емодзі.*

---

## 🛡️ Фаза 6: Аудит безпеки (Security Hardening Test)
Мета: Перевірити захист від несанкціонованого доступу та атак перебором.

- [ ] **Заборона доступу по паролю:** Спробуйте підключитися по SSH з локальної машини за допомогою пароля:
  ```bash
  ssh -o PreferredAuthentications=password root@<IP_сервера>
  ```
  *З'єднання має бути відхилено (`Permission denied`).*
- [ ] **Захист від брутфорсу (Fail2Ban):** Зробіть 6-7 спроб підключитися по SSH з неправильним користувачем/ключем.
  *Ваша IP-адреса має бути заблокована.*
  *(Щоб розблокувати: зайдіть через Cockpit UI -> Terminal -> виконайте `sudo fail2ban-client set sshd unbanip <ВАША_IP>`)*
- [ ] **Сканування портів:** Проскануйте сервер ззовні:
  ```bash
  nmap -Pn -p- <IP_сервера>
  ```
  *Повинні бути відкритими лише `22` (SSH), `80` (HTTP), `443` (HTTPS), `6443` (K3s API) та `9090` (Cockpit). Інші порти не повинні "світитися".*
