# 🧰 Адміністрування та Траблшутинг (G.E.N.A. K3s)

![Admin](https://img.shields.io/badge/Guide-Administration-8A2BE2?logo=gitbook&logoColor=white)
![UI First](https://img.shields.io/badge/Approach-UI_First-00C7B7?logo=react&logoColor=white)
![CLI](https://img.shields.io/badge/Fallback-CLI_Terminal-black?logo=gnome-terminal)

Філософія платформи **G.E.N.A.** полягає в тому, що 95% щоденних завдань, моніторингу та вирішення інцидентів мають виконуватися через зручні веб-інтерфейси сервісів. Ми залишаємо пряме підключення по SSH та консоль лише для найглибших системних збоїв рівня ядра ОС або коли мережева маршрутизація кластера повністю недоступна.

Цей посібник розділений за рівнями абстракції: від вбудованого автоматизованого конвеєра збірки (CI/CD) та GitOps до самого "заліза" сервера.

---

## 🛠 Рівень 1: Проблеми з CI/CD пайплайнами (Tekton)

Якщо ваш додаток не оновлюється або працює з помилками після зміни коду в Git, першим ділом йдемо в ArgoCD.

**Доступ:** `https://<Ваш_TEKTON_HOST>`

### Проблема: Пайплайн не запускається автоматично після пушу
* **Що означає:** Вебхук від вашого Git-провайдера (GitHub/GitLab) не зміг дістатися кластера або авторизуватися.
* **Рішення через UI:**
  1. Перейдіть у розділ **PipelineRuns**. Якщо списку запусків на момент вашого коміту немає — вебхук заблоковано на зовнішній стороні.
  2. Перевірте статус доставки вебхука в налаштуваннях вашого Git-репозиторію. Поширені помилки: неправильний домен, застарілий SSL-сертифікат або невідповідність токена WEBHOOK_TOKEN з файлу .env.

### Статус PipelineRun перейшов у стан Failed (Червоний)
* **Що означає:** Конвеєр запустився, але впав на одному з кроків (git-clone, build-and-push тощо).
* **Рішення через UI:**
  1. Клікніть на невдалий запуск у дашборді Tekton.
  2. Перегляньте логи конкретного кроку (Task). Якщо падає крок build-and-push (збирач Kaniko), шукайте синтаксичні помилки у вашому Dockerfile або перевірте, чи доступний локальний Docker-реєстр (registry.registry.svc.cluster.local:5000).

## 🛠 Рівень 2: Проблеми з розгортанням додатків (ArgoCD)

Коли образ успішно зібраний і відправлений в локальний реєстр, за його розгортання та відповідність декларативному маніфесту відповідає ArgoCD.

**Доступ:** `https://<Ваш_ARGOCD_HOST>`

### Проблема: Додаток у статусі `OutOfSync` (Жовтий)
* **Що означає:** Фактичний стан ресурсів у K3s-кластері не збігається з конфігурацією, що описана у вашому Git-репозиторії.
* **Рішення через UI:**
  1. Натисніть на картку додатку.
  2. Натисніть кнопку **APP DIFF** у верхньому меню — ви побачите детальне порівняння рядків коду маніфесту (як у GitHub), що саме відрізняється.
  3. Натисніть **SYNC**, щоб синхронізувати та примусово привести кластер до еталонного стану з Git.

### Проблема: Додаток у статусі `Degraded` або `Missing` (Червоний)
* **Що означає:** Ресурс Kubernetes не може бути коректно запущений через внутрішні помилки (OOMKilled, ImagePullBackOff, помилки синтаксису YAML).
* **Рішення через UI:**
  1. Відкрийте додаток в ArgoCD. У графічному дереві знайдете ресурс із червоним значком.
  2. Клікніть на нього та перейдіть у вкладку **Events** — там буде написана чітка причина відмови Kubernetes (наприклад, `Back-off restarting failed container`).
  3. Якщо новий коміт зламав продакшн, натисніть кнопку **HISTORY AND ROLLBACK**, оберіть попередню успішну версію та натисніть **Rollback**.

## 🐳 Рівень 3: Завислі контейнери та поди (Portainer)

Якщо ArgoCD показує, що ресурси створені, але додаток все одно не працює (наприклад, помилка в коді самого додатку), йдемо в Portainer.

**Доступ:** `https://<Ваш_PORTAINER_HOST>`

### Проблема: Pod у стані `CrashLoopBackOff` або `Error`
* **Що означає:** Ваш додаток запускається, одразу падає (через фатальну помилку в коді, відсутність бази даних або неправильні змінні середовища), і Kubernetes намагається його перезапустити нескінченно.
* **Рішення через UI:**
  1. Перейдіть у меню **Cluster** -> **Namespaces** і оберіть простір імен вашого додатку.
  2. Відкрийте розділ **Applications** (або Pods).
  3. Натисніть на іконку **📝 Logs** (Аркуш паперу) біля проблемного пода. Ви побачите консольний вивід вашого додатку і причину падіння (наприклад, `Connection refused to database`).
  4. Якщо треба покопатися всередині контейнера, натисніть іконку **>_ Console** (Термінал), оберіть `sh` або `bash` і натисніть Connect. Ви опинитесь всередині контейнера прямо в браузері!

### Проблема: Треба екстрено перезавантажити сервіс
* **Рішення через UI:** У розділі **Applications** виділіть ваш Deployment галочкою та натисніть кнопку **Restart** (або видаліть Pod — K3s миттєво створить новий, чистий екземпляр).

## 🗄️ Рівень 4: Локальне S3-сховище та Резервне копіювання (MinIO + Velero)

У версії 6.0.0 використовується повністю автономна in-cluster система збереження даних та конфігурацій.

**Доступ:** `https://<Ваш_MINIO_HOST>`

### Проблема: Збій виконання автоматичних бекапів Velero
* **Що означає:** Агент автоматичного копіювання Velero не може надіслати архівований стан кластера у сховище MinIO.
* **Рішення через UI:**
  1. Увійдіть в консоль керування **MinIO**, використовуючи логін та пароль адміністратора, задані у файлі .env через змінні MINIO_ROOT_USER та MINIO_ROOT_PASSWORD.
  2. Перевірте наявність та доступність бакета velero (він автоматично створюється скриптом mc-init під час розгортання платформи).
  3. Якщо сховище працює, але бекапи не записуються, проблема може бути у внутрішній маршрутизації мережі. Перевірте, чи зв'язаний Velero з правильним внутрішнім ендпоінтом сховища: http://minio-service.minio.svc.cluster.local:9000.

## 📊 Рівень 5: Аналіз логів та трафіку (Grafana + Loki)

Ви отримали алерт у **Signal**, що сайт повертає помилки `500 Internal Server Error`, або хочете знайти, хто намагається зламати ваш сайт.

**Доступ:** `https://<Ваш_GRAFANA_HOST>`

### Проблема: Треба знайти помилки в логах за певний час
* **Рішення через UI:**
  1. Відкрийте ліве меню -> **Explore**.
  2. У випадаючому списку джерел (зверху зліва) оберіть **Loki**.
  3. Використовуйте зручний візуальний конструктор (Label filters):
     * `namespace` = `ім'я_вашого_простору`
     * `app` = `ім'я_додатку`
  4. У полі пошуку (Operations) додайте фільтр `Line contains` і впишіть слово `Error` або `Exception`.
  5. Справа зверху оберіть часовий проміжок (наприклад, "Last 3 hours").

### Проблема: Не працює Ingress (веб-шлюз)
* **Рішення через UI:** Перейдіть у дашборд **NGINX Ingress Controller**. Перевірте графік **HTTP requests by status code**. Якщо бачите сплеск `502` або `503` — це означає, що Ingress працює, але ваш внутрішній додаток (Service/Pod) не відповідає або вимкнений.

## 🖥 Рівень 6: Проблеми рівня сервера/ОС (Cockpit)

Якщо інтерфейси Kubernetes туплять, сервер відповідає повільно, або прийшов алерт про 100% завантаження диска — йдемо в системну панель.

**Доступ:** `https://<IP_ВАШОГО_СЕРВЕРА>:9090`

### Проблема: Закінчилося місце на диску (Storage Full)
* **Рішення через UI:**
  1. Перейдіть у меню **Storage**.
  2. Перевірте, який саме розділ заповнений.
  3. Найчастіша причина у контейнерних середовищах — старі імеджи (образи). Перейдіть у вбудований **Terminal** в Cockpit і виконайте: `crictl rmi --prune` (видалити невикористовувані образи K3s).

### Проблема: Треба оновити сервер або перевірити системні логи
* **Рішення через UI:**
  1. Меню **Software Updates** дозволяє встановити патчі безпеки ОС в один клік.
  2. Меню **Logs** показує всі системні події. Фільтруйте за "Severity: Error", щоб знайти апаратні збої, проблеми з мережею або заблоковані через UFW/Fail2Ban IP-адреси.

## 🚨 Рівень 7: "Break Glass" (Консоль / CLI Fallback)

Використовується лише тоді, коли веб-інтерфейси (Ingress) повністю "впали" і ви не можете зайти в Grafana чи Portainer.

**Підключення:** `ssh user@<IP_ВАШОГО_СЕРВЕРА>`

### Базова діагностика кластера K3s
Якщо нічого не працює, перевіряємо головну службу:
```bash
sudo systemctl status k3s
# Якщо вона в стані failed, дивимось логи:
sudo journalctl -u k3s -f
# Перезапуск служби (безпечно для запущених контейнерів):
sudo systemctl restart k3s
```

### Пошук проблем через `kubectl`
Ви не можете зайти в UI, треба подивитися, що відбувається з кластером:
```bash
# Перевірка чи "жива" нода
kubectl get nodes

# Подивитись УСІ помилки у кластері за останній час (дуже корисна команда!)
kubectl get events -A --sort-by='.metadata.creationTimestamp' | grep -i warning

# Перевірити логи Ingress Controller (якщо сайти не відкриваються)
kubectl logs -n ingress-nginx -l app.kubernetes.io/name=ingress-nginx -f

# Перевірити статус випуску SSL сертифіката (якщо сайт пише "Небезпечно")
kubectl get certificates -A
kubectl describe certificate <ім'я_сертифіката> -n <namespace>
```

### Екстрене вимкнення брандмауера (тільки для дебагу!)
Якщо ви випадково заблокували собі доступ або підозрюєте, що мережеві правила UFW блокують трафік:
```bash
# Тимчасово вимкнути UFW
sudo ufw disable

# Переглянути заблоковані Fail2Ban IP-адреси (хто зараз у бані)
sudo fail2ban-client status sshd
```
> ⚠️ **Не забудьте увімкнути UFW назад після вирішення проблеми:** `sudo ufw enable`
