## 🖥️ 1. Управління сервером (Cockpit)

**Cockpit** — це ваш головний веб-інтерфейс для адміністрування базової операційної системи (Linux). 

* **Доступ:** `https://<IP_ВАШОГО_СЕРВЕРА>:9090`
* **Що тут можна робити:**
  * Перевіряти навантаження на CPU, RAM та диски на рівні ОС.
  * Керувати системними службами (Systemd).
  * Читати системні журнали (Journalctl), включаючи логи UFW (файрвола) та Fail2Ban.
  * Оновлювати пакети ОС без підключення по SSH.
  * Користуватися вбудованим веб-терміналом.

## 📦 2. Керування контейнерами (Portainer)

Якщо під час встановлення ви обрали **Portainer Server**, ви маєте зручний UI для візуального керування вашим Kubernetes-кластером (K3s).

* **Доступ:** `https://<Ваш_PORTAINER_HOST>`
* **Основні сценарії:**
  * Перегляд списку всіх розгорнутих Pod'ів, ConfigMap'ів, Secret'ів та Volume'ів.
  * Швидкий доступ до консолі (Shell) будь-якого контейнера прямо з браузера.
  * Читання логів окремих контейнерів у реальному часі.
  * Зміна ресурсних лімітів для тестових додатків.

> 💡 **Порада:** Portainer чудово підходить для *debugging* (пошуку помилок) та ручного керування, але для розгортання робочих додатків (Production) використовуйте **ArgoCD**.

## 🐙 3. Розгортання додатків (Docker + ArgoCD)

**ArgoCD** — це серце платформи. Він працює за принципом GitOps (інфраструктура як код).

Вам більше не потрібен зовнішній CI (напр. GitHub Actions). Кластер сам збирає ваші образи за допомогою **Tekton** та **Kaniko**.

### Етап 1: Налаштування Webhook у вашому Git-провайдері (GitHub/GitLab)
1. Після інсталяції в консолі ви побачили `Webhook URL` (напр. `https://tekton.yourdomain.com/webhook`) та `Webhook Secret`.
2. Зайдіть у налаштування вашого репозиторію -> Webhooks.
3. Вставте URL, оберіть Content type `application/json`, вставте Secret.
4. Тепер при кожному `git push` кластер автоматично запускатиме пайплайн `build-my-app-pipeline`. Цей пайплайн схиляється до вашого репозиторію, збере Docker-образ та відправить його у локальний реєстр: `registry.registry.svc.cluster.local:5000/apps/build-run-XXXX:latest`.

### Етап 2: GitOps-розгортання (ArgoCD)
1. У маніфестах вашого додатку (deployment.yaml) вкажіть образ: `registry.registry.svc.cluster.local:5000/apps/ваш-додаток:latest`.
2. В UI ArgoCD (`https://<Ваш_ARGOCD_HOST>`) створіть новий Application.
3. Вкажіть посилання на репозиторій з маніфестами. ArgoCD сам знайде збитий Tekton'ом образ і запустить його.

## 📊 4. Моніторинг та Логування (Grafana + Prometheus + Loki)

Усі метрики та логи автоматично збираються та візуалізуються в Grafana.

* **Доступ:** `https://<Ваш_GRAFANA_HOST>`
* **Навігація по дашбордах:**
  Перейдіть у меню **Dashboards**. Там вже встановлені готові панелі:
  * **Kubernetes Compute Resources:** моніторинг використання CPU/RAM вашими Pod'ами.
  * **Node Exporter Full:** детальна статистика по "залізу" вашого сервера.
  * **NGINX Ingress Controller:** статистика веб-трафіку (запити, помилки 404/5xx, час відповіді).
* **Пошук логів (Loki):**
  1. Перейдіть у меню **Explore**.
  2. Оберіть джерело даних (Data source) -> **Loki**.
  3. Використовуйте зручний конструктор запитів (Label filters), щоб знайти логи потрібного додатку, наприклад: `{namespace="production", app="my-backend"}`.

## 🗄️ 5. Робота зі сховищем (MinIO)

Якщо ви обрали "Velero + In-Cluster MinIO", у вас є власне S3-сховище.
* **Доступ:** `https://<MINIO_HOST>`.
* **Використання:** Там уже створено бакет `velero` для бекапів кластера. Ви можете створювати додаткові бакети для файлів ваших додатків (картинки, документи) і використовувати їх через S3 API, звертаючись за внутрішньою адресою: `http://minio-service.minio.svc.cluster.local:9000`.

## 🚨 6. Система алертів (Signal Месенджер)

Ваша платформа налаштована на проактивне сповіщення про проблеми. Замість того, щоб постійно дивитися в дашборди, система сама напише вам у **Signal**, якщо щось піде не так.

**Коли ви отримаєте повідомлення?**
* Сервер майже вичерпав вільне місце на диску (>85%).
* Високе навантаження на CPU тримається більше 10 хвилин.
* Будь-який Pod перейшов у стан `CrashLoopBackOff` (додаток постійно перезавантажується).
* Сайт/Ingress повертає велику кількість помилок `5xx HTTP`.

Сповіщення міститимуть назву проблеми, рівень критичності (Warning/Critical) та короткий опис. Коли проблема вирішиться, ви отримаєте сповіщення `[RESOLVED]`.

## 💻 7. Шпаргалка по CLI (kubectl)

Для тих, хто любить працювати через термінал SSH, K3s надає стандартний інструмент `kubectl`. Ось кілька базових команд для щоденної роботи:

```bash
# Переглянути всі поди у всіх просторах імен (namespaces)
kubectl get pods -A

# Переглянути споживання ресурсів (CPU/RAM) нодою та подами
kubectl top nodes
kubectl top pods -A

# Отримати детальний опис проблеми з конкретним подом
kubectl describe pod <pod-name> -n <namespace>

# Переглянути логи через CLI
kubectl logs <pod-name> -n <namespace>

# Перезапустити Deployment (якщо додаток завис)
kubectl rollout restart deployment <deployment-name> -n <namespace>
